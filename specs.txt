Minis Wettstetten - technische Zielarchitektur

Ziel:
Eine installierbare PWA fuer Ministranten-Dienstplanung, Chat, Statistik und
Benutzerverwaltung. Die App bleibt frontend-seitig schlank, produktive Daten
laufen aber ueber ein serverseitiges Gateway.

Produktivbetrieb:
- Hosting: Vercel.
- API: Vercel Node.js API Routes unter /api/*.
- Datenbank: Supabase Postgres im Free-Tier.
- Auth: Supabase Auth mit Passwort-Login und kostenlosen Standard-OAuth-
  Anbietern Google, GitHub und Microsoft. Apple bleibt explizites Opt-in nur
  mit Gebuehrenbefreiung oder akzeptierter Apple-Developer-Gebuehr.
- Secrets: nur in Vercel Runtime-Variablen, niemals in index.html.

Sicherheitsprinzipien:
- Least Privilege.
- Kein direkter Tabellenzugriff aus dem Browser.
- Supabase RLS aktiv, direkte Client-Policies verweigern Zugriff.
- App-Rollen werden serverseitig vor Datenzugriff geprueft.
- Dynamic UI-Ausgaben nutzen textContent/createTextNode, nicht innerHTML.
- CSP, SRI und Service-Worker-Scheme-Guards bleiben verpflichtend.

Datenmodell:
- public.app_state haelt den aktuellen App-Zustand als JSONB-Dokument.
- Root-Bereiche: users, publicProfiles, services, stats, chat.
- Private user-Daten werden nicht an normale Nutzer als Root exportiert.
- publicProfiles trennt Anzeigenamen von Verwaltungsdaten.

Rollen:
- user: eigene Teilnahme, Chat, eigenes Profil, eigene Statistik.
- admin: Dienstplanung, Statistikuebersicht.
- dev: Benutzerverwaltung, Backup, Backend-Diagnose.

Lokale Backends:
- mock: deterministische Tests ohne Netz.
- sqlite: lokaler Offline-Testbetrieb mit PBKDF2-Passwoertern.

Deployment:
- Vercel-Projekt mit Git-Integration.
- Supabase-Projekt in EU-Region, wenn verfuegbar.
- Runtime-Variablen aus .env.example.
- SQL-Migration aus supabase/schema.sql.
- OAuth-Redirects in Supabase und den Provider-Konsolen konfigurieren.
